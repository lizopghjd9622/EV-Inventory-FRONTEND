# 项目开发宪法（微信小程序前端版）

- Version: 2.0
- Ratified: 2026-03-03
- 适用范围：微信小程序 + H5 跨端前端项目（uni-app + Vue 3 + TypeScript）

本文件定义了本项目不可动摇的核心开发原则。所有 AI Agent 在进行技术规划和代码实现时，必须无条件遵循。

---

## 技术栈总览

| 层次 | 技术选型 | 说明 |
| ---- | -------- | ---- |
| 跨端框架 | **uni-app v3** | 同时支持微信小程序、H5，一套代码多端运行 |
| UI 框架 | **Vue 3 + Composition API** | `<script setup>` 语法，强制使用 |
| 类型系统 | **TypeScript 5.x（strict 模式）** | 所有代码必须有完整类型 |
| 构建工具 | **Vite + @dcloudio/vite-plugin-uni** | 开发体验与性能兼顾 |
| 状态管理 | **Pinia** | 替代 Vuex，更轻量、类型友好 |
| UI 组件库 | **uview-plus** | 专为 uni-app 设计，兼容微信小程序和 H5 |
| 网络请求 | **luch-request** | 支持拦截器，与 uni-app 生态深度集成 |
| 代码规范 | **ESLint + Prettier + Stylelint** | 统一代码风格，pre-commit 钩子强制检查 |
| 测试框架 | **Vitest + @vue/test-utils** | 单元测试与组件测试 |
| 工具函数 | **dayjs + lodash-es** | 日期处理与通用工具 |

## 第一条：简单性原则 (Simplicity First)

**核心：** 遵循“少即是多”哲学。绝不进行不必要的抽象，绝不引入非必需的依赖。

- **1.1 (YAGNI):** 只实现 `spec.md` 中明确要求的功能，不得为"将来可能用到"而提前抽象。
- **1.2 (组件粒度):** 组件应保持单一职责，不超过 300 行（含模板、脚本、样式）。超出时必须拆分。
- **1.3 (反过度工程):** 避免为简单状态引入复杂的状态管理。局部状态用 `ref`/`reactive`，跨页面状态才用 Pinia。
- **1.4 (依赖最小化):** 引入新的 npm 包前，必须确认无法用现有技术栈解决，并在 PR 中说明理由。

---

## 第二条：类型安全铁律 (Type Safety Imperative) - 不可协商

**核心：** TypeScript strict 模式下零 `any`，代码即文档。

- **2.1 (禁用 any):** **不可协商**：禁止使用 `any` 类型。使用 `unknown` + 类型守卫，或明确的联合类型。
- **2.2 (接口优于类型别名):** 对象结构定义优先使用 `interface`，联合/交叉类型等场景使用 `type`。
- **2.3 (API 类型同步):** 所有与后端交互的请求/响应类型，必须在 `src/types/api/` 下统一定义，不得在业务代码中内联定义。
- **2.4 (Props 必须声明类型):** 所有 Vue 组件的 props 必须使用 TypeScript 类型声明，禁止使用运行时 props 校验替代类型声明。

---

## 第三条：测试先行原则 (Test-First)

**核心：** 复杂业务逻辑和工具函数，必须有对应的 Vitest 测试覆盖。

- **3.1 (TDD 循环):** 对于纯函数、composables、store，遵循"Red-Green-Refactor"循环。
- **3.2 (状态验证优先):** 测试验证"结果状态"，而非"方法调用顺序"。禁止以 `spyOn` 的调用次数作为主要断言。
- **3.3 (组件测试):** 复杂交互组件必须有 `@vue/test-utils` 的组件测试，验证用户可观测的行为。

---

## 第四条：明确性原则 (Clarity and Explicitness)

**核心：** 代码的首要目的是让人类易于理解。

- **4.1 (错误处理):** **不可协商**：所有异步请求错误必须显式处理（`try/catch` 或 `.catch()`），不得静默忽略。统一通过 `luch-request` 拦截器处理通用错误（如 401、网络异常）。
- **4.2 (无隐式全局状态):** 禁止在 `setup()` 外部直接修改 Pinia store。所有状态变更必须通过 store 的 actions 进行。
- **4.3 (注释的意义):** 注释解释"为什么"而非"是什么"。所有导出的 composables 和工具函数必须有 JSDoc 注释。
- **4.4 (魔法数字):** 禁止在业务代码中出现未命名的魔法数字和魔法字符串。统一在 `src/constants/` 下定义命名常量或枚举。

---

## 第五条：移动端优先原则 (Mobile First)

**核心：** 以微信小程序体验为基准，兼顾 H5。

- **5.1 (性能优先):** 首屏加载时间是关键指标。禁止在首屏同步加载非必要资源。页面级组件使用路由懒加载。
- **5.2 (单位规范):** 样式尺寸统一使用 `rpx`（小程序端），H5 端通过 postcss 转换。禁止在布局中硬编码 `px`（边框 1px 除外）。
- **5.3 (平台差异隔离):** 平台差异代码（`#ifdef MP-WEIXIN` / `#ifdef H5`）必须集中在 `src/platform/` 或组件内的条件编译块中，禁止散落在业务逻辑里。
- **5.4 (图片优化):** 图片资源统一走 CDN，本地仅保留尺寸极小的 icon。禁止将大图资源提交到代码仓库。

---

## 第六条：单一职责原则 (Single Responsibility)

**核心：** 每个模块、每个文件、每个函数只做好一件事。

- **6.1 (组件职责):** 展示型组件（Presentational）只负责渲染，不调用 API、不操作 store。容器型组件（Container）负责数据获取和状态管理。
- **6.2 (Composable 内聚):** 每个 composable 封装一个独立的关注点（如 `useInfiniteScroll`、`useFormValidation`），不得将无关逻辑混入同一 composable。
- **6.3 (Store 模块化):** 每个业务域对应一个独立的 Pinia store 文件，禁止创建包含所有状态的"上帝 store"。

---

## 治理 (Governance)

本宪法具有最高优先级，其效力高于任何 `CLAUDE.md` 或单次会话中的指令。任何计划（`plan.md`）在生成时，都必须首先进行"合宪性审查"，确认技术选型与本宪法一致。
